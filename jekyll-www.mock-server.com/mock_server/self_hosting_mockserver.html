The fastest way to run your own MockServer is a single command. Pick whichever fits your environment — every option below starts a server that mocks and proxies HTTP and HTTPS on one port (default 1080):

# Docker — no Java required
docker run -p 1080:1080 mockserver/mockserver

# Node — no Java and no Docker (downloads a self-contained bundle)
npx -p mockserver-node mockserver run -p 1080

# Homebrew (macOS / Linux) — formula publishes with the next release
brew install mockserver && mockserver run -p 1080

Confirm it is up (the status endpoint returns 200 once MockServer is ready):

curl -i -X PUT http://localhost:1080/mockserver/status

This page is a task-oriented index of every self-hosting option. Each section links to the in-depth reference page where one exists.

Choose how to run MockServer:

  • Docker — one container, or a one-command docker compose recipe
  • Command-line / binary bundle — the mockserver CLI, run via Homebrew, Java, or a JVM-less Node bundle
  • Helm / Kubernetes — the OCI Helm chart
  • Executable JAR — run anywhere with Java 17+
  • Testcontainers — a throwaway container managed by your test run
  • Initialize & persist mocks — preload expectations or an OpenAPI spec, and persist them across restarts
  • Bootstrap from a browser HAR — turn recorded traffic into a running mock
 

Docker

Run the published image directly — it bundles Java and all dependencies, so nothing else needs installing:

docker run -p 1080:1080 mockserver/mockserver

Configure it with environment variables (any configuration property has an equivalent MOCKSERVER_* variable). For example, run it as a forwarding proxy:

docker run -p 1080:1080 \
    -e MOCKSERVER_PROXY_REMOTE_HOST=api.example.com \
    -e MOCKSERVER_PROXY_REMOTE_PORT=443 \
    mockserver/mockserver

See the full Docker container guide and the Docker download page for image variants, the AWS ECR Public mirror, and signature verification.

 

One-command docker-compose recipes

The repository ships task-oriented docker-compose.yml recipes — each is a self-contained directory you can docker compose up with no extra configuration. Clone the repo (or copy a recipe), cd into it, and run:

cd examples/docker-compose/mock-from-openapi
docker compose up
Recipe What it does
mock-from-openapi Serve mocks generated automatically from a mounted OpenAPI / Swagger spec.
record-replay-proxy Proxy to an upstream and record traffic to a replayable expectations file.
validation-proxy Proxy to an upstream and validate requests / responses against an OpenAPI spec.
chaos-proxy Proxy to an upstream while injecting latency and intermittent errors.

More configuration permutations (custom ports, mounted properties files, mTLS, persisted expectations) live in the docker-compose examples folder.

 

Command line & binary bundle

The mockserver command is the same whether it is installed by Homebrew, run from the executable JAR, or fetched on demand by the Node bundle. It uses subcommands:

Subcommand What it does Example
run Start MockServer in mock and/or proxy mode (the default — bare flags like -p 1080 imply run). mockserver run -p 1080
proxy Start as a forwarding proxy — shorthand for run --proxy-to. mockserver proxy --to api.example.com:8080 -p 1080
openapi Start and auto-create expectations from an OpenAPI spec. mockserver openapi ./petstore.yaml -p 1080
version Print the MockServer version and exit. mockserver version

The most useful run options:

Option Description
-p, --port <port> Port(s) to listen on; comma-separate to bind several, e.g. 1080,1081.
--proxy-to <target> Forward unmatched requests to host:port or a http(s)://host URL (a bare hostname with no port and no scheme is rejected).
--openapi <specUrlOrPath> Initialize expectations from an OpenAPI spec (file path or URL) on startup.
--init <fileOrGlob> Initialize expectations from a JSON file or glob on startup.
--persist <file> Persist expectations to the given file and reload them on restart.
-l, --log-level <level> Set the log level (e.g. INFO, DEBUG, WARN).

The full options reference (including the proxy and openapi subcommands, legacy flags, and configuration precedence) is on the Running From Command Line page.

 

Homebrew

On macOS or Linux, the mockserver command can be installed with Homebrew (the formula publishes with the next release):

brew install mockserver
mockserver run -p 1080
 

JVM-less binary bundle (no Java, no Docker)

If you have Node.js but neither Java nor Docker, the mockserver-node package can fetch a self-contained bundle — a trimmed Java runtime plus the server — for your platform on first use and run it:

npx -p mockserver-node mockserver run -p 1080
npx -p mockserver-node mockserver --openapi ./petstore.yaml -p 1080
npx -p mockserver-node mockserver --help

The bundle is downloaded from the MockServer GitHub Release, verified against its published SHA-256, and cached per-user. It exposes the full CLI (run, proxy, openapi, version, help). For corporate mirrors, cache locations, and proxy settings, see the mockserver-node documentation.

 

Helm / Kubernetes

The Helm chart is published to the GitHub Container Registry as an OCI artifact, so no helm repo add is needed:

helm upgrade --install --namespace mockserver --create-namespace --version {{ site.mockserver_version }} mockserver oci://ghcr.io/mock-server/charts/mockserver

See the Helm chart guide and the Kubernetes download page for values, proxy configuration, and signature verification.

 

Executable JAR

With Java 17 or higher installed, download the standalone jar from Maven Central and run it — the same CLI as above applies:

java -jar mockserver-netty-no-dependencies-{{ site.mockserver_version }}.jar run -p 1080

Get the jar from the downloads page (or directly from Maven Central), and see Running From Command Line - Using Java for more.

 

Testcontainers

To have a MockServer container started and torn down automatically by your test run, use MockServer with Testcontainers rather than self-hosting a long-lived instance.

 

Initialize & persist mocks

However you run MockServer, you can preload expectations at startup and persist them across restarts. These work as CLI flags, configuration properties, or environment variables:

What you want CLI flag Environment variable
Preload expectations from a JSON file or glob --init <file> MOCKSERVER_INITIALIZATION_JSON_PATH
Preload expectations from an OpenAPI spec --openapi <specUrlOrPath> MOCKSERVER_INITIALIZATION_OPENAPI_PATH
Persist expectations to a file across restarts --persist <file> MOCKSERVER_PERSIST_EXPECTATIONS=true + MOCKSERVER_PERSISTED_EXPECTATIONS_PATH

For example, with Docker, mount a spec and serve it as a mock:

docker run -p 1080:1080 \
    -v $(pwd)/openapi.yaml:/config/openapi.yaml \
    -e MOCKSERVER_INITIALIZATION_OPENAPI_PATH=/config/openapi.yaml \
    mockserver/mockserver

See Initializing Expectations, Persisting Expectations, and the configuration properties page (each property lists its environment-variable form) for full details.

 

Bootstrap from a browser HAR

To get a working mock from real traffic in seconds: open your browser DevTools, record the requests on the Network tab, and Export HAR. Then import that file into a running MockServer — it creates one expectation per recorded entry:

curl -X PUT "http://localhost:1080/mockserver/import?format=har" \
    --data-binary @recording.har

The same endpoint also imports Postman collections. See Importing Expectations from HAR & Postman for format detection, options, and examples.
